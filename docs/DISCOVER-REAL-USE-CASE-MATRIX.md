# Discover real use-case matrix

Discover's production flagship is open-source contribution funding intelligence. Every row begins with activity that already happens on GitHub and ends only where RESOLVE has a durable record of the economic outcome.

| Workflow | Primary user | Trigger | Existing manual process | Pain | Real input | RESOLVE analysis | Human decision | Financial action | Recipient | Repeat loop | Completion proof |
|---|---|---|---|---|---|---|---|---|---|---|---|
| Repository funding review | Foundation, grants lead, or ecosystem operator | A scheduled or operator-requested GitHub snapshot changes | Export repository activity, compare spreadsheets, ask maintainers, and reconcile grants manually | Reviews, documentation, support, and release work disappear behind merge counts | Persisted public GitHub repository snapshot and event-level Evidence records | Classify accepted work, compare activity with active program rules, expose uncovered categories and source freshness | Decide whether the gap deserves investigation or an existing program should change | Start a persisted Mission or open the real operating program | Contributors represented by source identities | Return after the next verified snapshot and inspect the deterministic delta | Snapshot fingerprint, scan time, evidence URLs, persisted Mission/program reference |
| Recognition-debt review | Program operator or maintainer | Accepted activity has no matching active policy | Search PRs/issues and funding history by hand | Work is visible but has no recognition mechanism | GitHub activity records, ProgramVersion/PolicyVersion, PaymentAuthorization history | List uncovered work records, contributors, prior funding, identity readiness, and exact missing mechanism | Inspect proof, resolve identity, or prepare a policy decision in Mission | No direct payout from Discover; routing goes to Mission/Profile/Communities | Overlooked contributor after identity and policy checks | Recompute when activity, policy, or identity state changes | Evidence record plus later obligation/authorization/receipt lineage |
| Maintainer concentration review | Foundation director, security lead, or grants team | Review, release, or security work becomes concentrated | Rely on anecdotal maintainer knowledge | Bus-factor risk is hard to quantify and easy to overstate | Accepted event counts grouped by public GitHub actor | Show the numerator, denominator, selected contributors, categories, and time window | Decide whether to investigate support or succession funding | Start a persisted Mission with the selected repository and evidence IDs | Maintainers whose observed work supports the calculation | Compare the same calculation on each snapshot | Source event list and transparent concentration formula |
| Funding coverage review | Funder or community operator | Verified obligations approach or exceed confirmed pool capital | Compare multiple dashboards and transaction records | Funding shortfalls and recipient blockers are discovered late | Active program, confirmed fund stakes, checkpoint state, obligations, identities, payout destinations | Separate verified obligations, pool balance, shortfall, eligible recipients, blocked recipients, and last distribution | Open the operating program, resolve payout setup, or review authorization in Capital | Funding occurs through the existing program/Capital flow, never directly from an estimate | Eligible obligation recipients | Recompute after deposits, identity changes, authorization, or settlement | Persisted funding intent/stake, confirmed chain state, receipt |
| Outcome proof review | Funder, operator, contributor, or judge | A settlement batch is confirmed | Reconcile transaction explorer, spreadsheet, and payout list manually | It is difficult to prove why money moved and who received it | Confirmed SettlementBatch, ChainTransaction, Receipt, obligations, and evidence lineage | Present opportunity, program, recipients, amount, confirmation time, receipt, and exact transaction hash | Inspect public proof | None; this is a read-only proof surface | Confirmed settlement recipients | New proof appears only after authoritative confirmation | Persisted receipt and exact Arc transaction URL |

## Production gate

A workflow remains hidden or in a truthful setup state when its row cannot be completed from current persisted records. Research, music, video, and creator domains are not promoted on the production Discover surface by this implementation.
