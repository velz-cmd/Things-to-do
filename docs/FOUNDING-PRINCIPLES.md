# RESOLVE — Founding Principles (Frozen)

Source: *Distribution Bootstrap for Payments Founders* (Parts 1–4).  
**Status:** Immutable. Engineering work must not violate this document.

---

## 0. What this is

This is a **distribution thesis** first. Payments are what travels through inherited distribution.

Settlement **consumes attribution** that ecosystems already record. We do not create creator economies — **we discover them.**

**Product heuristic (start here, not at payments):**

> Where is value already being created that nobody settles?

Not: "How do we pay creators?"

---

## 1. The one law

**Every successful idea steals distribution from software that already exists.**

Rails are solved. Distribution is the problem.  
Do not build audiences, platforms, marketplaces, or social networks. **Attach.**

---

## 2. Everything solved except settlement

Navidrome · Owncast · Immich · PeerTube · RSSHub · MusicBrainz · Git · ActivityPub — each solved its domain.

**Nothing needs replacing. Only settlement is missing.**

| Forbidden | Required |
|-----------|----------|
| Replace · fork · compete · convince · migrate users | Attach permissionlessly |

Integration shapes: plugin · sidecar · wrapper · middleware · federation peer.

---

## 3. Metadata was always settlement-grade

Attribution metadata was **secretly payment input for years** — used only for display.

MusicBrainz · EXIF · `attributedTo` · RSS author · git blame — **payment graphs already built.**

We consume them as **settlement input**, not invent new graphs. AI scores within upstream attribution.

---

## 4. Value → authorization → settlement

Sequence:

```
Value exists (consumption / contribution)
  → Software records it
  → Metadata identifies payees
  → Authorization recognizes amount owed
  → Settlement transfers money (later, batched)
```

**Authorization** = economic fact at event time (engineering term — not "rights").  
**Settlement** = fulfillment when capital exists.

Spotify model: play immediately increases owed; money moves later.

Founders/operators **fund fulfillment** — they do not create the fact of value.

Copy: **"You've earned $X. Settlement pending funding."** — not "Founder decided to reward you."

---

## 5. Three moats (together)

1. **Distribution Connectors** — install beside upstream (plugin, sidecar, …)  
2. **Attribution Registry** — existing identity keys → payout destination  
3. **Attribution + usage + settlement history** over years — impossible to recreate cold  

---

## 6. Invisible infrastructure

Settlement Core is **plumbing** (like DNS/TLS) — not the brand users remember.

External: current integrations (GitHub today; Navidrome, PeerTube, Owncast, Jellyfin, RSSHub, Mastodon upcoming).  
**No public connector numbering** — no hierarchy implied.

Workspace = **operational interface** to the Settlement Core (evidence, authorization, funding fulfillment).

---

## 7. Aadi's test

1. Which **consumption event**?  
2. Which **upstream metadata field** is settlement input?  
3. **Inherits distribution** without new users?  
4. **Permissionless**?  
5. **Authorize at event time**, settle later?

---

## 8. Doctrine (one paragraph)

RESOLVE does not create creator economies—it discovers them. Mature ecosystems emit economic events and store attribution. We attach permissionlessly, interpret metadata as settlement-grade data, **authorize at consumption**, and fulfill through portable batched settlement. Settlement Core is invisible infrastructure; Distribution Connectors inherit communities; Attribution Registry resolves identities without inventing graphs. The long-term asset is **attribution + usage + settlement history** across ecosystems—not the rail.
