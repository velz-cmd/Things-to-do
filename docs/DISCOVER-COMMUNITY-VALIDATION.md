# Discover community validation

## Target community

The first implementation is designed for maintainers, foundations, and funding operators working with public GitHub repositories. `navidrome/navidrome` is the initial integration-shaped reference because its community already produces durable contribution activity on GitHub and is also named in the supplied Distribution Bootstrap research. The product supports other persisted public repository snapshots without claiming that every repository has adopted RESOLVE.

## Current workflow and pain

- Existing behavior: contributors open and merge pull requests, review work, resolve issues, update documentation, publish releases, support users, and handle security work on GitHub.
- Existing manual workflow: operators inspect GitHub activity, funding platforms, identity records, and payout spreadsheets separately.
- Product hypothesis: accepted non-code work and recipient blockers are easy to overlook; program coverage and funding readiness are expensive to reconcile repeatedly.
- Available source data: public GitHub repository/PR/review/issue/release data, persisted RESOLVE Evidence, program policy versions, obligations, funding stakes, settlement batches, chain transactions, and receipts where they genuinely exist.
- Workflow replaced: repeated manual comparison of contribution activity, program rules, funding state, identities, and settlement proof.

## Validation status

External community validation has **not yet been completed** for this redesign. No maintainer interview, foundation operator session, or grants-team usability session is recorded in the repository. Consequently:

- Feedback received: none recorded.
- What a community understood without explanation: not yet measured.
- What a community attempted to use: not yet measured.
- Real testnet funding or payout action from an external community: not verified by this implementation.
- Production-readiness claim: blocked on external validation and an authenticated, configured Arc/Circle staging run.

## Required validation session

1. Ask a maintainer or funding operator to select their repository without explaining the product model.
2. Ask them which work their current program misses and which contributor is blocked.
3. Observe whether they can trace the evidence and explain the concentration calculation.
4. Ask them to start a Mission or open the real program without assistance.
5. Where configured and authorized, fund a testnet program through the existing Capital/Communities flow.
6. Verify submitted versus confirmed state, recipient attribution, the public receipt, and the exact transaction.
7. Record direct quotes only with permission; do not convert assumptions into feedback.
